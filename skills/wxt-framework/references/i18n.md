# WXT + i18n

WXT handles `_locales/` as a public asset directory and respects `default_locale` in the manifest. The mechanism is the same as plain MV3 — see the `extension-i18n` skill for the full pattern. This page covers the WXT-specific bits.

## Where `_locales/` lives

```
<project>/
└── public/
    └── _locales/
        ├── en/
        │   └── messages.json
        ├── es/
        │   └── messages.json
        └── ...
```

WXT copies `public/` verbatim into `.output/<browser>-mv3/`. So `public/_locales/en/messages.json` ends up at `.output/chrome-mv3/_locales/en/messages.json`.

## Manifest setup

In `wxt.config.ts`:

```ts
manifest: {
  name: '__MSG_extension_name__',
  description: '__MSG_extension_description__',
  default_locale: 'en',
  action: { default_title: '__MSG_action_title__' },
}
```

`__MSG_<key>__` references are resolved by Chrome at install time based on the user's locale.

## In TS/TSX code

```ts
const name = chrome.i18n.getMessage('extension_name');
const greeting = chrome.i18n.getMessage('greeting', ['Alice']);
```

WXT auto-imports `chrome.*` if you enable auto-imports in config; otherwise it's the standard `chrome.i18n` global.

A small typed helper for components:

```ts
// src/lib/i18n.ts
export const t = (key: string, substitutions?: string | string[]) =>
  chrome.i18n.getMessage(key, substitutions);
```

```tsx
<h1>{t('extension_name')}</h1>
<p>{t('items_count', [String(n)])}</p>
```

## Type-safe message keys

WXT can generate types for your message keys via a build hook:

```ts
// wxt.config.ts
import fs from 'node:fs';
import path from 'node:path';

export default defineConfig({
  hooks: {
    'build:before': async () => {
      const msgs = JSON.parse(
        fs.readFileSync('public/_locales/en/messages.json', 'utf8')
      );
      const keys = Object.keys(msgs).map((k) => `'${k}'`).join(' | ');
      const out = `// Auto-generated. Do not edit.\nexport type MessageKey = ${keys || 'never'};\n`;
      fs.writeFileSync(path.join('src', 'i18n-keys.d.ts'), out);
    },
  },
});
```

Then update your helper to use `MessageKey`:

```ts
import type { MessageKey } from '../i18n-keys';

export const t = (key: MessageKey, substitutions?: string | string[]) =>
  chrome.i18n.getMessage(key, substitutions);
```

Now `t('typo_key')` is a TypeScript error.

## Crowdin integration

Crowdin is the most popular translation platform for browser extensions (used by Bitwarden, Vimium C). Setup:

`crowdin.yml`:

```yaml
project_id: "12345"
api_token_env: CROWDIN_TOKEN
preserve_hierarchy: true
files:
  - source: /public/_locales/en/messages.json
    translation: /public/_locales/%two_letters_code%/messages.json
```

Workflow:

1. Push: `crowdin push` uploads source strings.
2. Translators do their work in Crowdin.
3. Pull: `crowdin pull` downloads translations and writes them back to `public/_locales/{locale}/messages.json`.
4. Commit, build, ship.

Automate via GitHub Actions:

```yaml
# .github/workflows/i18n.yml
on:
  schedule:
    - cron: '0 6 * * *'   # daily 6am UTC
  workflow_dispatch:
jobs:
  pull-translations:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: crowdin/github-action@v2
        with:
          upload_sources: true
          download_translations: true
          create_pull_request: true
        env:
          CROWDIN_TOKEN: ${{ secrets.CROWDIN_TOKEN }}
          GITHUB_TOKEN: ${{ github.token }}
```

## Weblate integration

Self-hosted alternative used by Privacy Badger (EFF). Configure Weblate with the GitHub integration:

1. Add `public/_locales/en/messages.json` as the source file pattern.
2. Set `public/_locales/{locale}/messages.json` as the translation pattern.
3. Weblate opens PRs as translations land.

## RTL languages

For Arabic/Hebrew/Farsi/Urdu, your popup/options pages should support RTL. WXT doesn't auto-detect; do it in code:

```ts
// entrypoints/popup/main.tsx (or wherever your popup mounts)
const lang = chrome.i18n.getUILanguage();
const isRTL = ['ar', 'he', 'fa', 'ur'].includes(lang.split('-')[0]);
document.documentElement.dir = isRTL ? 'rtl' : 'ltr';
document.documentElement.lang = lang;
```

Then in CSS:

```css
[dir="rtl"] .popup-container {
  text-align: right;
}
```

Or use Tailwind's `rtl:` variants.

## Pluralization

`chrome.i18n` has no ICU MessageFormat support. The standard pattern is per-form keys:

```json
{
  "items_zero": { "message": "No items" },
  "items_one":  { "message": "1 item" },
  "items_many": { "message": "$COUNT$ items", "placeholders": { "count": { "content": "$1" } } }
}
```

```ts
function itemsMsg(n: number) {
  if (n === 0) return t('items_zero');
  if (n === 1) return t('items_one');
  return t('items_many', [String(n)]);
}
```

For complex pluralization (Russian, Arabic) consider adopting a library like FormatJS — but that's overhead. Per-form keys cover 90% of cases.

## CI lint

The plugin's `extension-i18n/references/check-i18n.sh` script verifies:

- Every locale has the same key set as `en/`.
- Every `getMessage('foo')` call references a defined key.
- Every key has a `description` field for translators.

Run in CI:

```yaml
- name: i18n lint
  run: bash chrome-extension-builder/skills/extension-i18n/references/check-i18n.sh .
```

## Caveats

- `__MSG_*__` placeholders only work in **user-visible** manifest fields (`name`, `description`, `action.default_title`, etc.). They don't work in technical fields like `permissions` or `host_permissions`.
- `chrome.i18n.getMessage` is synchronous, but it requires `_locales/` to be present in the loaded extension. In dev mode, `wxt dev` includes them automatically; in tests, mock `chrome.i18n.getMessage`.
- Don't ship empty `messages.json` for a locale — Chrome falls back to `default_locale` if a key is missing within a locale, but an empty file is treated as "no translations available."
